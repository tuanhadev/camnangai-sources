## What is Vibe Coding (Nope, Don't skip this)

Before diving into what's wrong, we need to define what vibe coding actually is. The problem is that everyone has a different definition:

- **Type A:** Doing most work via plain English prompts while still reading the code
- **Type B:** Ignoring code entirely and only prompting
- **Type C:** Non-developers writing code via AI
- **Type D:** Using autocomplete in IDEs

What's your definition?

## The Problem

Now, based on your definition of vibe coding, you will judge if it's worth it or not, and that's the wrong way. Let me explain.

If your definition is around Type B, Type C, and you want to build a software that can be used by real users, sorry to say, but you will never be able to do it.

Vibe coding meant for Type B, Type C should be that — You can build a basic prototype of what you have in your mind, or small tools for your own use case. If you are expecting more than that, it's your fault, not Vibe Coding's fault.

Type A and Type D are not really vibe coders; they are using AI to write code. If you know how to write code and use that knowledge while taking the help of AI, you are not vibe coding. Stop telling people that you vibe-coded this app.

## What is Term coding

In Term coding, *term* means terminology or vocabulary.

Let's suppose there are 2 people:

- **A** — A developer with 6 months of experience.
- **B** — A founder of a big marketing company

Now, let's give 2 challenges to both of them.

### 1st Challenge

Both person (A, B) needs to build an authentication system for a website using AI.

**Person A will ask AI to:**

> Build a secure authentication system for a web app with email/password login and Google OAuth. Include:
>
> **Email/Password Auth**
> - Sign-up, login, logout, password reset, and email verification.
> - Hash passwords using Argon2id (or bcrypt if not available).
> - Enforce basic password rules and use salted, constant-time comparison.
> - Store sessions in secure HTTP-only cookies (SameSite=Lax/Strict).
>
> **Google Login**
> - Implement Google OAuth 2.0 / OIDC.
> - Store minimal user info and support linking Google accounts to existing email accounts.
>
> **Security**
> - Use prepared statements/ORM to avoid SQL injection.
> - Validate and sanitize all inputs.
> - Add rate limiting to login/reset endpoints.
> - Protect against CSRF, XSS, and session fixation.
> - Use env variables for secrets (DB, JWT, OAuth client secret).
> - Use HTTPS and reject insecure requests in production.
>
> **Output Requirements**
> - Provide folder structure, DB schema, and key API endpoints.
> - Write clean, modular code with comments explaining critical logic.

**Person B will ask AI:**

> "Please create a login system for a website that lets people sign in with email or Google. Make sure it works with any errors."

What do you think, who will win? Who will get better results, which actually help you build a professional website?

Most of us say it's obvious that Person A will win because they have experience, but did you notice — he did not use the experience; he used the **terminology/vocabulary**, while Person B doesn't know the vocabulary.

If I reverse the challenge, you will see a completely different result.

### 2nd Challenge

Both person (A, B) needs to build an AD video using AI.

Person A will ask for a general AD video with a 1 or 2-line prompt.

Person B will ask with a detailed prompt with all the specifications.

Now, who will win? **Person B**, right?

Why? Not because he is a smart person, but because he has the terminology and vocabulary to describe what he gained in marketing.

> Don't confuse this with Prompt Engineering. You can be a great prompt writer, but still fail because you don't know the exact terms and vocabulary used to build that stuff.

Meaning, if you want to succeed in vibe coding or any fields where you want to take advantage of AI, you need to have a dictionary of words to describe what you want because it's generative AI.

Generative AI generates based on the input you give, and as we all know — **"Garbage in, Garbage out."**

Remember:

> "A bad coach can make a good team lose."

You are a coach, and your team is AI. You need a vocabulary and words that your team understands.
